# ESP32 组合秤与屏幕端 RS485 JSON 通讯计划书

本文档定义了主控端 (`esp32_sorter`) 与屏幕端 (`esp32_sorter_screen`) 之间的 RS485 JSON 通讯协议及其实施计划。双方项目均需严格遵守此文档定义的协议格式与交互逻辑。

---

## 1. 通讯架构概述

*   **物理层**: RS485 串口半双工通讯。
*   **波特率**: 建议 115200 或 460800，以保证 JSON 文本传输的低延迟。
*   **拓扑结构**: **主从轮询模式 (Master-Slave)**。
    *   **主机 (Master)**: 主控端 (`esp32_sorter`)。
    *   **从机 (Slave)**: 屏幕端 (`esp32_sorter_screen`)。
*   **防干扰机制 (CRC校验)**: 由于工业现场电磁干扰严重，协议采用类 NMEA 格式对 JSON 进行包裹校验。
    *   **格式**: `$<纯JSON文本>*<CRC8校验和>\n`
    *   示例: `$ {"req":"dashboard"} *A5\n`
*   **交互机制 (一问一答)**:
    1.  **主机发起**: 主控根据自身空闲时间片，周期性 (如 10Hz) 向屏幕发送当前的状态数据包。
    2.  **从机应答**: 屏幕收到合法数据后，立刻（如在 10ms 内）返回包含当前屏幕状态、所处页面及用户点击事件的应答包。
    3.  **超时处理**: 如果主控超时未收到屏幕应答，不重发，而是直接进入下一个轮询周期，发送最新的数据。

---

## 2. 协议定义 (草案)

### 2.1 主机发往从机：请求与数据更新 (Master -> Slave)

主机除了下发数据外，也隐式地在向从机“要”事件。由于主机并不知道从机当前在哪个页面，可以默认下发核心主页数据，或者根据上一轮从机应答中反馈的 `current_page` 动态切换下发内容。

#### 1. 下发生产主界面数据 (Dashboard)
```json
$ {"type":"dashboard","data":{"speed":120,"yield":1250.5,"capacity":300.0}} *[CRC]\n
```

#### 2. 下发系统维护数据 (Admin - Encoder/Laser/Cutter)
```json
$ {"type":"admin_encoder","data":{"pulse_count":102456,"velocity":1.5,"status":1}} *[CRC]\n
```

### 2.2 从机发往主机：状态与事件上报 (Slave -> Master)

从机收到主机的包后，必须立刻回复。回复的内容主要是它的**内部状态**和**刚刚发生的事件**。

#### 1. 常规应答 (无特殊事件，仅上报当前页面)
告诉主机“我现在在哪个页面”，以便主机下一轮发对数据。
```json
$ {"current_page":"dashboard","events":[]} *[CRC]\n
```

#### 2. 包含点击事件的应答
如果用户在两次轮询之间点击了按钮，屏幕将其缓存，并在下一次被点名时一起发给主机。
```json
$ {"current_page":"dashboard","events":[{"cmd":"tare"}]} *[CRC]\n
```

#### 3. 包含调试参数的事件应答
```json
$ {"current_page":"admin_cutter","events":[{"cmd":"test_cutter","params":1500}]} *[CRC]\n
```

## 3. 主控端 (`esp32_sorter`) 实施计划 (Master)

1.  **调度权控制**: 
    *   在主控的主循环或专属任务中，以固定频率 (如 100ms) 触发一次通讯。
    *   这样做的好处是**通讯完全不会打断主控的硬实时任务（如称重、落料控制）**。
2.  **发送逻辑**:
    *   读取上一轮记录的屏幕页面 `last_known_screen_page`。
    *   根据此页面，打包对应的 JSON 数据。
    *   计算 CRC8，加上 `$` 和 `*\n` 包裹后通过 RS485 发送。
3.  **接收逻辑**:
    *   发送后开启一个短超时 (如 20ms) 监听回复。
    *   若收到 `$ ... *\n`，校验 CRC。若通过，则剥离 JSON 并用 `ArduinoJson` 解析。
    *   提取 `current_page` 更新内部记录。
    *   遍历 `events` 数组，执行对应的系统命令（如置零、启动电机）。

---

## 4. 屏幕端 (`esp32_sorter_screen`) 实施计划 (Slave)

1.  **事件缓存机制**:
    *   因为屏幕变成了从机，不能立刻发数据。必须建立一个轻量级的事件队列。
    *   用户点击屏幕（触发 `LV_EVENT_CLICKED`）时，将事件指令压入队列。
2.  **被动应答任务**:
    *   在一个极高优先级的任务中监听 RS485 RX。
    *   一旦检测到完整的 `$<JSON>*<CRC>\n` 且校验通过，立即：
        1.  将接收到的数据分发给 `SystemContext` 刷新界面。
        2.  **打包应答数据**: 获取当前 Tab 页名称，弹出缓存队列中的所有事件。
        3.  **发送应答**: 生成 JSON，加上 CRC 外壳，立刻通过 RS485 返回。
